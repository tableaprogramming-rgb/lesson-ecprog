SECTION 1: What is Git and GitHub?

Git vs GitHub

These are often confused—they're different but related:

Git GitHub
A version control system software that runs on your computer A website/service that hosts Git repositories in the cloud
Tracks changes to your code over time (history) Stores your code online and enables collaboration
You use Git commands in your terminal/command prompt You interact with GitHub through a web browser or Git commands
Free, open-source Free for public repositories, paid plans for private repos
Example: git commit, git push Example: https://github.com/username/repo-name

Why Use Git and GitHub?

  • Version History: Keep a complete history of changes. You can go back to any previous version if needed.
  • Collaboration: Multiple people can work on the same project without overwriting each other's work.
  • Backup: Your code is safely stored in the cloud, not just on your local computer.
  • Portfolio: Employers and teachers can see your work and contributions on GitHub.
  • Learning: You can browse and learn from other people's code repositories.

SECTION 2: Installing and Setting Up Git

Step 1: Install Git

Windows:

  1. Go to https://git-scm.com/download/win
  2. Download and run the installer
  3. Accept all default options during installation
  4. Open Command Prompt or PowerShell after installation

macOS:

  1. Open Terminal (Applications → Utilities → Terminal)
  2. Run: brew install git (requires Homebrew)
  3. Or go to https://git-scm.com/download/mac and download the installer

Linux:

  1. Open Terminal
  2. Run: sudo apt-get install git (Ubuntu/Debian) or sudo yum install git (RedHat/CentOS)

Step 2: Verify Git Installation

Open your terminal/command prompt and type: git --version

Expected output: You should see the Git version number (e.g., "git version 2.40.0")

If you see version information, Git is installed successfully. If you see "command not found," restart your terminal and try again.

Step 3: Configure Git with Your Name and Email

Git needs to know who you are. Type these commands in your terminal:

git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

Replace "Your Name" with your actual name and "your.email@example.com" with your email.

Why? Every commit (change you save) will be tagged with your name and email, so people know who made the change.

To verify your configuration was saved, run: git config --global --list

SECTION 5: Clone the Repository to Your Computer

What Does "Clone" Mean?

Cloning downloads your repository from GitHub to your local computer so you can work on it.

Step 1: Get the Repository URL

  1. Go to your repository on GitHub (https://github.com/YOUR-USERNAME/[project-name])
  2. Click the green "Code" button
  3. Copy the URL (it should start with https://github.com/...)

Step 2: Clone the Repository

  1. Open your terminal/command prompt
  2. Navigate to where you want to store your project (e.g., Documents or Desktop)
  3. Run this command (paste the URL you copied): 
    git clone https://github.com/YOUR-USERNAME/[project-name]
  4. Expected output: A new folder with your repository name will be created with all repository files

Step 3: Enter Your Repository Folder

Navigate into your cloned repository by typing the folder name:

cd [project-name]

You're now inside your project folder. Any Git commands you run will affect this repository.

SECTION 6: The Basic Git Workflow

Understand the 4-Step Process

Every time you make changes to your code, follow this workflow:

Step 1: Check What Changed (Status)

Why: See which files you've modified since the last commit.

Step 2: Prepare Changes (Stage)

Why: Tell Git which files you want to save in the next commit. You don't have to commit all changes at once.

Step 3: Save Changes (Commit)

Why: Create a permanent snapshot of your code at this point. This becomes part of your version history.

Step 4: Push to GitHub (Upload)

Why: Upload your commits to GitHub so they're backed up in the cloud.

Visual Diagram:

Your Computer (Local)              GitHub (Remote/Cloud)
┌──────────────────────┐            ┌──────────────────────┐
│ Your Files (Working) │            │ Repository (Online)  │
│  - [source files]    │    PUSH    │ - [source files]     │
│  - [header files]    │     →→→    │ - [header files]     │
│  - [other code]      │    PULL    │ - README.md          │
│  - README.md         │     ←←←    │ - (version history)  │
└──────────────────────┘            └──────────────────────┘
         ↑ COMMIT
         │ (saves to local history)
         │
    ┌────┴────────────────┐
    │ Staging Area       │
    │ (files to commit)  │
    └────────────────────┘

SECTION 7: Complete Workflow Example

Scenario: You Just Finished Writing Your Source Code

Step 1: Check Repository Status

Run this command in your terminal (inside your repository folder):

git status

Output explains: Which files are new (untracked), modified, or unchanged. Red files are not yet staged.

Step 2: Stage Your Changes

Add specific files to the staging area:

git add [filename]

OR add all changes at once:

git add .

What this does: Tells Git "these are the files I want to save in my next commit"

Step 3: Check Staging Status

Verify what you're about to commit:

git status

Expected output: Files in green are staged and ready to commit. Files in red are not staged yet.

Step 4: Commit with a Message

Create a snapshot of your staged changes:

git commit -m "Add feature implementation"

Important: Use meaningful commit messages! Describe WHAT changed and WHY.

Step 5: Push to GitHub

Upload your commits from your local computer to GitHub:

git push

Expected output: You should see a message like "1 file changed" or similar. Your code is now on GitHub!

Step 6: Verify on GitHub

  1. Go to your repository on GitHub (https://github.com/YOUR-USERNAME/[project-name])
  2. Refresh the page
  3. You should see your files appear in the repository!
  4. Click on a file to see its contents
  5. Click "commits" to see your commit history

SECTION 8: Essential Git Commands Cheat Sheet

Command What It Does Example
git status Show which files changed since last commit Check before committing
git add [filename] Stage a specific file for commit git add main.c
git add . Stage all changed files at once Quick way to stage everything
git commit -m "[message]" Save staged changes with a description git commit -m "Add feature"
git push Upload commits to GitHub After committing, push to backup
git log View your commit history See all changes you've made
git pull Download latest changes from GitHub Use if working on multiple computers

SECTION 9: How to Write Good Commit Messages

Why Commit Messages Matter

  • Documentation: Future you will thank present you when reviewing what changed
  • Collaboration: Other developers (and your instructor) can understand your work
  • Debugging: When finding a bug, you can read commit history to see what changed

Examples of Good vs Bad Commit Messages

BAD GOOD
"update" "Add core function implementation for feature X"
"fix bug" "Fix null pointer exception when handling edge case Y"
"changes" "Implement function Z with proper error handling"
"stuff" "Add README with project description and setup instructions"
"asdf" "Refactor module A to handle empty/null cases"

Commit Message Format

Simple template:

[Verb] [What changed] [Why it's important]

Example commit messages:

  • "Initialize project structure with core data types and main logic"
  • "Implement core operation/function for primary functionality"
  • "Add error handling and edge case validation"
  • "Write comprehensive README with usage instructions"

SECTION 10: Managing Your Project Repository

Folder Structure on GitHub

Organize your files clearly. Suggested structure:

[project-name]/
├── README.md              (Description of your project)
├── src/                   (All source code)
│   ├── [main source files]
│   ├── [header files]
│   └── [additional modules]
├── obj/                   (Compiled files - listed in .gitignore)
├── bin/                   (Executable files - listed in .gitignore)
└── .gitignore             (Tells Git which files to ignore)

What NOT to Commit

Your .gitignore file (created when you set up the repo) tells Git to ignore:

  • Compiled object files - (.o, .obj, .class, etc.) - can be regenerated
  • Executable files - (.exe, .out, etc.) - can be regenerated
  • Libraries - (.a, .lib, .so, etc.) - can be regenerated
  • obj/ or build/ folders - Directories with intermediate build files
  • bin/ or dist/ folders - Directories with executables

Why? These are generated files. Only commit source code (program files) and configuration files (README, Makefile, etc.). This keeps your repository clean and small.

Update Your README.md

Your repository needs a README explaining your project. Click "Edit" on your README.md on GitHub or edit locally.

Good README includes:

  • Project title: Clear name describing what the project is
  • Brief description: What is it? Why did you build it?
  • How to compile/build: Instructions for setup (gcc, javac, python, npm, etc.)
  • How to run: Instructions to execute your program or tests
  • Features: What your program can do
  • Requirements: What needs to be installed (compiler, runtime, libraries, etc.)

SECTION 12: Common Issues & How to Fix Them

Issue 1: "fatal: not a git repository"

Problem: You're trying to run git commands outside your repository folder.

Solution: Use "cd" to navigate into your [project-name] folder, then try again.

Issue 2: "error: src refspec main does not match any"

Problem: You're trying to push but haven't committed anything yet.

Solution: Commit your files first using "git add" and "git commit", then "git push".

Issue 3: "Please tell me who you are" when committing

Problem: Git doesn't know your name/email.

Solution: Run the setup commands from Section 2, Step 3.

Issue 4: Changes not showing on GitHub after pushing

Problem: You forgot to stage or commit before pushing.

Solution: Run "git status" to check. Then "git add", "git commit", and "git push" in that order.

Issue 5: "Your branch is ahead of 'origin/main' by X commits"

Problem: You've committed locally but haven't pushed to GitHub yet.

Solution: Run "git push" to upload your commits.

Issue 6: Can't remember what files you changed?

Solution: Run "git status" to see. Run "git log" to see your entire commit history.

SECTION 13: Best Practices & Next Steps

Committing Regularly

  • Don't wait until the end: Commit after completing each function
  • Small commits are better: One commit per feature is ideal
  • Frequent pushes: Push to GitHub regularly as backup

Example Commit Timeline

  1. Commit 1: "Initialize project structure with basic setup"
  2. Commit 2: "Implement core feature/function 1"
  3. Commit 3: "Implement core feature/function 2"
  4. Commit 4: "Implement core feature/function 3"
  5. Commit 5: "Implement remaining functions and features"
  6. Commit 6: "Add user interaction and main program flow"
  7. Commit 7: "Add comprehensive README with setup instructions"

Remember: This is just an example. Your commits should match the actual work you did. One commit per feature or logical change is ideal.

Advanced Tips (Optional)

  • Create branches: For experimenting with features without affecting main code
  • Create issues: Track bugs and features to implement
  • Use tags: Mark important releases (v1.0, v2.0, etc.)
  • Collaborate: Add collaborators to work together on the same project

   Visit https://github.com/YOUR-USERNAME/[project-name]

If Something Goes Wrong

Check status:
   git status

See all commits:
   git log

See what changed in a file:
   git diff filename

   git checkout filename

Undo changes (after staging):
   git reset filename
